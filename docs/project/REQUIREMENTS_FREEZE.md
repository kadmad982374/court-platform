## تجميد المتطلبات للإصدار الأول

> هذه الوثيقة **مجمّدة** في نهاية المرحلة 0.
> أي تغيير لاحق يستلزم تحديثًا صريحًا هنا، مع إشارة في `PROJECT_ASSUMPTIONS_AND_DECISIONS.md`.

## 1) المتطلبات الوظيفية المجمّدة (FR)

### FR-ORG — الهيكلية التنظيمية
- FR-ORG-01: دعم 14 فرعًا ثابتًا (المذكورة في الوثيقة الوظيفية §3.2).
- FR-ORG-02: داخل كل فرع 4 أقسام ثابتة: الصلح، البداية، الاستئناف، التنفيذ.
- FR-ORG-03: ربط المحاكم بالفرع وبنوع القسم، مع دعم الغرف عند اللزوم.

### FR-IDN — الهوية والحسابات
- FR-IDN-01: تسجيل دخول، خروج، تجديد جلسة.
- FR-IDN-02: نسيان كلمة المرور عبر رقم الموبايل وكود تحقق وإعادة إرساله.
- FR-IDN-03: إنشاء حساب يتضمن: اسم المستخدم، الفرع، المحكمة، رقم الموبايل.
- FR-IDN-04: قراءة بيانات المستخدم الحالي (`/users/me`).

### FR-AC — ضبط الوصول
- FR-AC-01: أدوار تقنية حصرية: CENTRAL_SUPERVISOR, BRANCH_HEAD, SECTION_HEAD, ADMIN_CLERK, STATE_LAWYER, READ_ONLY_SUPERVISOR, SPECIAL_INSPECTOR.
- FR-AC-02: عضوية المستخدم في قسم/فرع (UserDepartmentMembership).
- FR-AC-03: منح/حجب صلاحية الدخول إلى محاكم محددة لكل محامٍ من قبل رئيس القسم.
- FR-AC-04: تفويض صلاحيات رئيس القسم للموظف الإداري كاملًا أو جزئيًا.
- FR-AC-05: قاعدة ملكية المحامي تمنع وصول محامٍ لدعاوى محامٍ آخر.
- FR-AC-06: قواعد حدود الفرع والقسم مفروضة في الخادم.

### FR-LR — قيد الدعوى الجديدة
- FR-LR-01: إنشاء دعوى جديدة مع: الجهة العامة، صفتها (مدعية/عليها)، الخصم، المحكمة، الغرفة، رقم الأساس، تاريخ القيد الأول (ثابت لا يتغير)، تاريخ الجلسة الأولى، سبب التأجيل الأول.
- FR-LR-02: تعديل البيانات الأساسية فقط من قبل رئيس القسم/الموظف الإداري ضمن النطاق.
- FR-LR-03: إسناد الدعوى إلى محامي دولة من نفس القسم وله صلاحية المحكمة.
- FR-LR-04: استثناءً: فصل مباشر دون إحالة في حالات خاصة محددة.

### FR-LP — متابعة الجلسات
- FR-LP-01: ترحيل جلسة بإدخال: تاريخ الجلسة القادمة + سبب التأجيل الجديد.
- FR-LP-02: حفظ تاريخ الجلسة السابقة وسبب التأجيل السابق دائمًا.
- FR-LP-03: قائمة أسباب تأجيل معيارية أولية (تبليغ الأطراف، إخطار الأطراف، كشف وخبرة، جواب الإدارة، جواب الخصم، تدقيق، تبليغ بالصحف، إخطار بالصحف) قابلة للتوسعة المضبوطة.
- FR-LP-04: عدم فقدان البيانات السابقة عند أي تحديث (سجل ترحيل بنمط append-only).
- FR-LP-05: شاشة جلسات يوم محدد.

### FR-DF — الفصل وتسجيل القرار
- FR-DF-01: حقول الفصل: رقم القرار، تاريخه، نوعه، المبلغ، العملة، الخلاصة.
- FR-DF-02: نوع القرار حصرًا من: FOR_ENTITY, AGAINST_ENTITY, SETTLEMENT, NON_FINAL.
- FR-DF-03: المرحلة بعد الفصل تصبح FINALIZED وتدخل سجل الفصل وفق شهر القرار.
- FR-DF-04: تصحيح محدود بعد الفصل لرئيس القسم، مع أرشفة.

### FR-RR — سجل الفصل الشهري
- FR-RR-01: عرض شهري بحسب شهر تاريخ القرار.
- FR-RR-02: حقول العرض: الجهة، الخصم، المحكمة، الأساس/السنة، القرار، تاريخه، نوعه، المبلغ، الخلاصة.
- FR-RR-03: مشتق منطقيًا من بيانات الفصل، لا مصدر حقيقة مستقل.

### FR-ST — الانتقال بين المراحل
- FR-ST-01: نقل دعوى من الصلح/البداية إلى الاستئناف ينشئ CaseStage جديدة من نوع APPEAL مرتبطة بـ parent.
- FR-ST-02: المرحلة السابقة تصبح read-only بعد الانتقال.
- FR-ST-03: نقل إلى التنفيذ ينشئ ExecutionFile مستقلًا مرتبطًا بالـ LitigationCase ومرحلة المصدر.

### FR-EX — التنفيذ
- FR-EX-01: قيد ملف تنفيذي بحقول: الجهة المنفذة، المنفذ عليه، نوع الملف، رقمه، السنة.
- FR-EX-02: إجراءات تنفيذية مؤرخة (action_date, action_type, action_description) مع مرفقاتها.
- FR-EX-03: لا جلسات داخل التنفيذ.

### FR-AT — المرفقات
- FR-AT-01: رفع صور و PDF لمرفق على نطاق: CASE_STAGE أو EXECUTION_FILE أو EXECUTION_ACTION.
- FR-AT-02: تحقق checksum SHA-256 وحجم ونوع المحتوى.
- FR-AT-03: استمرار ظهور المرفقات للقراءة عند انتقال المرحلة.

### FR-RM — التذكيرات
- FR-RM-01: المحامي يضيف تذكيرات بدعواه بتاريخ وساعة، مع نص.
- FR-RM-02: حالات: PENDING, DONE, CANCELLED.

### FR-NT — الإشعارات
- FR-NT-01: إشعار عند قيد أي دعوى جديدة لأصحاب العلاقة.
- FR-NT-02: تحترم الملكية والنطاق التنظيمي.

### FR-LL — المكتبة القانونية
- FR-LL-01: تصنيف وتصفح وبحث بالكلمات المفتاحية.
- FR-LL-02: محتوى يديره قسم الدراسات.

### FR-PD — دليل الجهات العامة
- FR-PD-01: تصنيف بحسب الوزارات/الهياكل العليا، مع تصفح وبحث.

### FR-CR — القرارات والتعاميم
- FR-CR-01: قرارات وتعاميم وزارة العدل وقرارات وتعاميم الإدارة قابلة للاستعراض والبحث.

### FR-RP — التقارير
- FR-RP-01: تقارير للإدارة المركزية، رئيس الفرع، رئيس القسم، المحامي — كلٌ ضمن نطاقه.

### FR-AU — التدقيق
- FR-AU-01: تسجيل الأحداث المذكورة في §12 من الوثيقة التقنية، مع actor/action/entity/timestamp/branch_scope/details.

### FR-UI — الواجهة
- FR-UI-01: واجهة عربية RTL بسيطة.
- FR-UI-02: لوحة رئيسية حسب الدور.
- FR-UI-03: شاشات: تسجيل الدخول/الحساب، جلسات اليوم، قيد دعوى، قائمة الدعاوى، تفاصيل الدعوى، سجل الفصل، الملفات التنفيذية، المرفقات، التذكيرات، الإشعارات، المكتبة، دليل الجهات، القرارات والتعاميم، التقارير.

## 2) المتطلبات التقنية المجمّدة (TR)

- TR-01: **Backend** = Java 21 + Spring Boot 3 (Web, Data JPA, Security, Validation) + PostgreSQL + Flyway + Springdoc OpenAPI + Lombok + Testcontainers.
- TR-02: **Frontend** = React 18 + TypeScript + Vite + React Router + TanStack Query + Axios + React Hook Form + Zod + Tailwind CSS + shadcn/ui.
- TR-03: **نمط المعمارية** = Modular Monolith، فصل صارم بين Domain / Application / Infrastructure / API.
- TR-04: 17 وحدة تقنية أساسية (انظر `DOMAIN_BOUNDARIES.md`).
- TR-05: قاعدة بيانات علائقية واحدة، مخططات مفصولة منطقيًا أو schemas حسب الحاجة.
- TR-06: ترقيمات الـ APIs تحت `/api/v1/...` كما في الوثيقة التقنية §10.
- TR-07: الصلاحيات تُفرض في طبقة الـ Application/Service وليس في الـ UI.
- TR-08: التدقيق Audit مطلوب على الأحداث الحساسة في كل وحدة.
- TR-09: حالات الكيانات الأساسية كما في §7 من الوثيقة التقنية.
- TR-10: الـ Frontend RTL، arabic-first، role-aware.

## 3) داخل نطاق الإصدار الأول (In-Scope)
1. الهيكلية التنظيمية (الفروع، الأقسام، المحاكم).
2. الهوية وضبط الوصول والصلاحيات (بما فيها صلاحيات المحاكم والتفويض).
3. قيد الدعوى وتعديل البيانات الأساسية والإسناد.
4. ترحيل الجلسات وأسبابها وأرشفتها.
5. الفصل وتسجيل القرار.
6. سجل الفصل الشهري كعرض مشتق.
7. الانتقال إلى الاستئناف.
8. الملفات التنفيذية وإجراءاتها.
9. المرفقات.
10. التذكيرات.
11. الإشعارات (الحد الأدنى: قيد دعوى جديدة).
12. المكتبة القانونية ودليل الجهات العامة والقرارات والتعاميم.
13. التقارير الإشرافية الأساسية.
14. التدقيق Audit.
15. الواجهة الأمامية العربية وفق الأدوار.

## 4) خارج نطاق الإصدار الأول (Out-of-Scope حاليًا)
- التكامل الفعلي مع البريد الإلكتروني (FR مستقبلي، يُراعى تصميميًا فقط).
- التكامل المباشر مع أنظمة المحاكم/الوزارات الخارجية.
- توقيع إلكتروني/PKI.
- تطبيقات Mobile Native منفصلة (الواجهة المتجاوبة كافية).
- تحليلات BI متقدمة وDashboards تفاعلية معقدة.
- ترجمة متعددة اللغات (النظام عربي بالكامل).
- Workflow engine عام أو محرك قواعد ديناميكي.
- إدارة عقود/فوترة/محاسبة.
- نظام رسائل داخلية (chat) بين المستخدمين.
- إدارة وثائق متقدمة (DMS) خارج نطاق المرفقات الأساسية.

## 5) نقاط احتملت تفسيرًا والقرار المعتمد
> التفاصيل في `PROJECT_ASSUMPTIONS_AND_DECISIONS.md`. ملخصها هنا:

| الموضوع | التفسير المعتمد |
|---|---|
| سجل الفصل | عرض مشتق (Read Model) من بيانات الفصل، ليس جدولًا منفصلًا. |
| نقل الاستئناف | CaseStage جديدة بنوع APPEAL مرتبطة بـ parent_stage_id؛ السابقة read-only. |
| نقل التنفيذ | ExecutionFile مستقل مرتبط بـ LitigationCase وبـ source_stage_id؛ ليس CaseStage. |
| الموظف الإداري | يبدأ بنسخة كاملة من صلاحيات رئيس القسم، يحجبها رئيس القسم عبر UserDelegatedPermission. |
| الفصل المباشر دون إحالة | يقتصر على رئيس القسم في حالات خاصة محددة، ويُسجَّل في Audit. |
| المرحلة السابقة | غير قابلة للتعديل بعد الانتقال؛ المرفقات والجلسات تبقى مقروءة. |
| التحقق من رقم الموبايل | عبر كود OTP مع زر إعادة إرسال (تطبيق المزود لاحقًا). |
| اللغات | عربي فقط في الإصدار الأول. |
